# KREP-022: Default Managed Resources Status

## Problem statement

When users create instances of CRDs generated by ResourceGraphDefinitions, they
have no built-in way to see which Kubernetes resources that instance owns. The
only way to discover managed resources today is to manually query resources by
kro labels across the cluster — a poor experience for debugging, auditing, and
understanding instance state.

While `state` and `conditions` tell you *whether* an instance is healthy, they
don't tell you *what* it manages. Users frequently ask: "what did this instance
create?" and "why is it stuck?" — both require resource-level visibility that
the current status doesn't provide.

### Use cases

1. **Debugging**: A `WebApplication` instance is stuck in `IN_PROGRESS`. The
   user needs to see which specific resources are pending or errored.
2. **Auditing**: A platform team wants to verify that a `DatabaseCluster`
   instance created the expected resources (StatefulSet, Services, ConfigMaps).
3. **Tooling**: A UI or CLI tool wants to show a tree view of all resources
   managed by an instance without needing cluster-wide label queries.
4. **Collections**: A `WorkerPool` using `forEach` creates N pods. The user
   wants to see all N resources and their individual states.

## Proposal

### Overview

Add a new default status field `managedResources` to every CRD generated by an
RGD. This field lists all graph nodes and their concrete resource references,
updated on every reconciliation cycle. It sits alongside the existing `state`
and `conditions` defaults and follows the same opt-out pattern.

### Design details

#### Status schema

The `managedResources` field is an array of objects, one per graph node
(excluding the instance node itself and external reference nodes). Each entry
contains the node's identity, its current reconciliation state, and references
to the actual Kubernetes resources it manages. External nodes (`External`,
`ExternalCollection`) are excluded because they are read-only references, not
resources managed by the instance:

```yaml
status:
  state: ACTIVE
  conditions: [...]
  managedResources:
    - nodeID: "deployment"
      apiVersion: "apps/v1"
      kind: "Deployment"
      nodeType: "Resource"
      state: "SYNCED"
      graphRevision: 3
      resources:
        - name: "myapp-deploy"
          namespace: "default"
    - nodeID: "worker-pods"
      apiVersion: "v1"
      kind: "Pod"
      nodeType: "Collection"
      state: "SYNCED"
      graphRevision: 3
      resources:
        - name: "myapp-worker-0"
          namespace: "default"
        - name: "myapp-worker-1"
          namespace: "default"
        - name: "myapp-worker-2"
          namespace: "default"
```

**Field descriptions:**

| Field | Type | Description |
|-------|------|-------------|
| `nodeID` | string | The graph node identifier from the RGD `resources[].id` |
| `apiVersion` | string | API group and version (e.g. `apps/v1`, `v1` for core) |
| `kind` | string | Resource kind |
| `nodeType` | string | One of: Resource, Collection |
| `state` | string | Node reconciliation state: SYNCED, IN_PROGRESS, ERROR, SKIPPED, DELETING, DELETED, WAITING_FOR_READINESS, INCLUDED, EXCLUDED |
| `graphRevision` | integer | RGD generation this node was last reconciled against (optional; allows visibility into rolling updates between RGD revisions) |
| `total` | integer | Total observed resource count for this node (omitted when not truncated) |
| `truncated` | boolean | Whether the `resources` list was capped (omitted when false) |
| `resources` | array | List of concrete resource references (may be truncated) |
| `resources[].name` | string | Resource name |
| `resources[].namespace` | string | Resource namespace (empty for cluster-scoped) |

#### CRD schema integration

The field is added as a default in `pkg/graph/crd/defaults.go` alongside
`defaultStateType` and `defaultConditionsType`. It is registered in both
`newCRDSchema()` and `SetCRDStatus()` using the same `if _, ok` guard pattern,
meaning:

- It is only added when `statusFieldsOverride` is true (the default).
- Users can define their own `managedResources` status field in the RGD schema
  to override the default (same as `state` and `conditions`).

```go
if _, ok := status.Properties["managedResources"]; !ok {
    status.Properties["managedResources"] = defaultManagedResourcesType
}
```

#### Population during reconciliation

The `managedResources` field is populated in `initialStatus()` during every
status update, using data already available in the reconciliation context:

1. **Node list**: `rcx.Runtime.Nodes()` returns all graph nodes in topological
   order (instance node and external nodes excluded).
2. **Node metadata**: `node.Spec.Meta` provides ID, GVR (group/version/resource),
   Type (Resource, Collection), and Namespaced flag.
3. **Node state**: `rcx.StateManager.NodeStates[nodeID]` provides the current
   reconciliation state (SYNCED, ERROR, etc.).
4. **Resource references**: `node.GetObserved()` returns the observed
   `[]*unstructured.Unstructured` from which name and namespace are extracted.
5. **Graph revision**: `rcx.Runtime.GraphRevision()` (or equivalent) provides the
   RGD generation the graph was built from. This value is set on each managed
   resource entry, making it visible when different nodes are at different
   revisions during a rolling update.

A new `GetObserved()` getter is added to `runtime.Node` to expose the private
`observed` field. GVK is derived from `node.Spec.Template.GroupVersionKind()`.

**Performance**: No additional API calls are needed. The data is assembled from
objects already fetched during the reconciliation cycle. For a typical RGD with
5-10 nodes, this adds approximately 500 bytes to the status payload.

#### Truncation

Collections can produce large numbers of resources (e.g. a `forEach` over
hundreds of items). To bound the status payload size, each node entry's
`resources` array is capped at a configurable maximum. When truncated, a
`truncated` boolean is set to `true` and a `total` integer reports the actual
count, so consumers know the list is incomplete.

```yaml
- nodeID: "worker-pods"
  apiVersion: "v1"
  kind: "Pod"
  nodeType: "Collection"
  state: "SYNCED"
  graphRevision: 3
  total: 500
  truncated: true
  resources:
    - name: "myapp-worker-0"
      namespace: "default"
    # ... up to MaxManagedResourceEntriesPerNode entries
```

The limit is configured via `RGDConfig.MaxManagedResourceEntriesPerNode`, which
defaults to 100. This can be tuned per controller deployment. When the observed
count for a node is within the limit, `truncated` and `total` are omitted to
keep the common case clean.

**Note:** `MaxManagedResourceEntriesPerNode` is distinct from
`RGDConfig.MaxCollectionSize`. `MaxCollectionSize` limits how many resources
are *actually created* by a collection node (a runtime concern), while
`MaxManagedResourceEntriesPerNode` limits how many resource references are
*reported in status* (a display concern). A collection could have 50 items
(within `MaxCollectionSize`) but status might truncate to 20 entries if
configured lower. In practice, if `MaxCollectionSize` is already bounded,
truncation may rarely trigger — but the safeguard is still needed for future
changes or very large bounded collections.

The `total` field is always an integer representing the full count of observed
resources for that node, regardless of truncation. This allows tooling to show
"5/500 resources" without needing cluster-wide queries to determine the actual
count.

#### Reserved field handling

The `managedResources` key is added to the skip list in `updateStatus()` (alongside
`conditions` and `state`) so that user-defined CEL status expressions cannot
accidentally overwrite it.

#### kubectl integration

The `managedResources` data is accessible via `kubectl get <instance> -o yaml`
or `kubectl get <instance> -o jsonpath='{.status.managedResources}'`. A
priority-1 printer column could be added in the future to show managed resource
count with `kubectl get -o wide`, but JSONPath limitations in Kubernetes make
this non-trivial and is out of scope for this KREP.

## Other solutions considered

### Labels-only approach

Leave the status as-is and rely on label queries
(`kubectl get all -l kro.run/instance-name=<name>`).

**Why not**: Requires cluster-wide permissions, doesn't show per-node state,
doesn't work with standard `kubectl get <instance>` workflows, and is
invisible to tools consuming the CRD status.

### Separate Status CRD

Create a companion `InstanceStatus` CRD per RGD instance to avoid bloating the
main resource status.

**Why not**: Adds operational complexity (extra CRDs, extra watches, garbage
collection). The `managedResources` payload is small (hundreds of bytes for
typical RGDs) and fits naturally in the existing status subresource.

### Events-only approach

Emit Kubernetes Events for resource creation/deletion instead of status fields.

**Why not**: Events are ephemeral (default TTL 1 hour), not queryable, and
don't provide a point-in-time view of what an instance currently manages.
Events are complementary but not a replacement.

## Scoping

### What is in scope for this proposal?

- New `defaultManagedResourcesType` OpenAPI schema in `defaults.go`
- Registration in `newCRDSchema()` and `SetCRDStatus()` with opt-out guard
- `GetObserved()` getter on `runtime.Node`
- `buildManagedResources()` helper in instance controller
- Integration into `initialStatus()` and `updateStatus()`
- Unit tests for schema, population, and edge cases
- Integration test verifying end-to-end behavior

### What is not in scope?

- **Printer columns**: Showing managed resource count in `kubectl get` output
  is desirable but requires JSONPath support for array aggregation, which
  Kubernetes doesn't natively support. May come in a follow-up.
- **Resource health details**: Per-resource condition forwarding (e.g., showing
  a Deployment's `Available` condition) would add significant payload size and
  complexity. The node-level `state` is sufficient for the initial implementation.
- **Historical tracking**: The field shows current state only, not a log of
  previously managed resources.
- **CLI integration**: A `kro status <instance>` command that pretty-prints the
  managed resources tree is out of scope but would be a natural follow-up.

## Testing strategy

### Requirements

- Standard Go test infrastructure (`go test`)
- envtest for integration tests (already set up in `test/integration/`)
- No additional clusters or external resources needed

### Test plan

**Unit tests:**

1. `pkg/graph/crd/defaults_test.go`:
   - Verify `defaultManagedResourcesType` schema structure validity
   - Verify `newCRDSchema()` includes `managedResources` when `statusFieldsOverride=true`
   - Verify `managedResources` is NOT added when `statusFieldsOverride=false`
   - Verify user-defined `managedResources` is not overwritten

2. `pkg/controller/instance/status_test.go`:
   - Test `buildManagedResources()` with single-resource nodes
   - Test with collection nodes (multiple observed objects)
   - Test that external ref nodes are excluded from output
   - Test with skipped/errored nodes (empty resources list)
   - Test `graphRevision` is set correctly on each entry
   - Verify `initialStatus()` includes `managedResources` key
   - Verify `updateStatus()` does not overwrite `managedResources` from user CEL

3. `pkg/runtime/node_test.go`:
   - Test `GetObserved()` returns nil before `SetObserved()`
   - Test `GetObserved()` returns correct list after `SetObserved()`

**Integration tests:**

1. Create an RGD with multiple resource types (Deployment, Service, ConfigMap)
2. Create an instance and verify `status.managedResources` is populated
3. Verify resource references match actual cluster resources
4. Verify node states match expected reconciliation outcomes
5. Delete the instance and verify cleanup

## Discussion and notes

- The `managedResources` field name was chosen over alternatives like
  `ownedResources` (implies ownership semantics that may conflict with
  OwnerReferences from KREP-004) and `resourceInventory` (too generic).
- External nodes (`External`, `ExternalCollection`) are excluded from
  `managedResources` because they are read-only references, not resources
  managed by the instance. Including them would be misleading — the instance
  does not create, update, or delete external resources. Users who need
  visibility into external references can inspect the RGD definition directly.
- `INCLUDED` and `EXCLUDED` states reflect `includeWhen` evaluation outcomes,
  giving users visibility into conditional resource inclusion. `EXCLUDED`
  indicates a node whose `includeWhen` condition evaluated to false and was
  therefore not created.
